ภาพรวมเกมเพลย์และกลไกหลัก (Game Overview)
"Synapse Shield: Toxic Ingress" เป็นเกมบนเว็บเบราว์เซอร์แนว Orbital Deflector / Physics Action ที่ออกแบบมาให้เล่นง่ายแต่เสพติดสูง โดยเน้นไปที่ทักษะปฏิกิริยาตอบสนองความเร็วสูง (Twitch Gameplay) ผู้เล่นจะรับหน้าที่ปกป้อง "เซลล์สมอง (Core)" ที่อยู่กึ่งกลางหน้าจอ ด้วยการควบคุม "โล่พลังงาน" ที่วิ่งหมุนโค้งรอบวงกลมตามตำแหน่งเคอร์เซอร์เมาส์แบบ 100% เกมเพลย์ขับเคลื่อนด้วยกลไกความเสี่ยงและผลตอบแทน (Risk/Reward) ผู้เล่นต้องใช้เมาส์ลากโล่ไป "กระแทกหรือปัด" สัญลักษณ์สารพิษที่พุ่งเข้ามาให้กระเด็นออกไป และต้อง "เปิดช่องโหว่" ชั่วคราวเพื่อให้ Core ดูดซับสารอาหารดีๆ ที่ลอยเข้ามาเพื่อฟื้นฟูพลังชีวิต ตัวเกมยกระดับความน่าตื่นเต้นด้วยการใส่ "Game Juice" อย่างพิถีพิถัน ทั้งระบบฟิสิกส์เวกเตอร์สะท้อน (Elastic Collision) ที่อิงตามแรงเหวี่ยงหรือการสะบัดเมาส์ (Flick Momentum) ผสานกับเอฟเฟกต์หน้าจอสั่น (Screen Shake) และอนุภาคแสงแตกกระจาย ทำให้ทุกการปะทะเกิดความรู้สึกที่ทรงพลังและสะใจ

สถาปัตยกรรมทางเทคนิคฝั่งหน้าบ้าน (Client-Side Architecture 99%)
เพื่อให้เกมรองรับการเล่นที่ลื่นไหลระดับ 144Hz+ บน Google Chrome โดยไม่เกิดอาการภาพกระตุก (Micro-stutter) ตัวเกมถูกสร้างขึ้นบนโครงสร้างแบบ Hybrid React-Canvas ซึ่งแบ่งแยกหน้าที่การทำงานอย่างเด็ดขาด ดังนี้:

    React (The Manager): ทำหน้าที่เป็นเปลือกนอก ดูแลเฉพาะส่วน UI ติดต่อผู้ใช้ เช่น หน้าเมนูหลัก ตัวเลขคะแนน และหน้า Game Over

    Data Handling (Zero Re-render): ถือเป็นหัวใจหลักของความลื่นไหล โดยเกมจะปฏิเสธการใช้ useState ของ React ในการเก็บพิกัดวัตถุโดยเด็ดขาด เพื่อป้องกันการ Re-render หน้าจอซ้ำๆ แต่จะใช้ useRef เป็นหน่วยความจำหลัก (Game State) ในการเก็บข้อมูลตัวแปรทุกอย่างในเกม

    HTML5 Canvas (The Worker): ประมวลผลภาพและฟิสิกส์ภายในลูป requestAnimationFrame ซึ่งจะปรับจังหวะการคำนวณและการวาดภาพให้ซิงก์กับอัตรารีเฟรชของจอผู้เล่นโดยอัตโนมัติ

    Custom Physics & Optimization: ทิ้งระบบฟิสิกส์หนักๆ ของ Game Engine ทั่วไป แล้วใช้คณิตศาสตร์บริสุทธิ์แทน การหมุนของโล่คำนวณผ่านมุมองศาด้วยฟังก์ชัน Trigonometry อิงจากตำแหน่งพิกัด X/Y ของเมาส์โดยตรง (Zero Input Latency) การชนใช้กลไก Circle-to-Circle Collision ที่เบาบาง ผสานกับการใช้ Object Pooling (คลังรีไซเคิลวัตถุ) เพื่อหมุนเวียนศัตรูและเอฟเฟกต์ ลดการเกิด Garbage Collection ที่ทำให้เกมหน่วง

สถาปัตยกรรมฝั่งหลังบ้านและความปลอดภัย (Backend Architecture 1%)
เซิร์ฟเวอร์หลังบ้านถูกออกแบบมาให้เป็น Lightweight API ซึ่งจะรับหน้าที่แค่จัดการฐานข้อมูลตารางคะแนน (Leaderboard) เท่านั้น จะไม่มีการประมวลผลระบบฟิสิกส์ใดๆ บนเซิร์ฟเวอร์ การเชื่อมต่อจะเกิดขึ้นเป็น Asynchronous POST Request เพียงครั้งเดียวเมื่อจบเกม (Game Over) เพื่อลดภาระโหลดของเครือข่าย อย่างไรก็ตาม เพื่อป้องกันผู้เล่นใช้หน้าต่าง DevTools แอบแก้คะแนนบนเบราว์เซอร์ ระบบจึงเสริมการป้องกันด้วย Lightweight Anti-Cheat โดยฝั่งหน้าบ้านจะต้องส่งตัวเลข "ระยะเวลาที่เล่นจริง (Playtime)" ไปคู่กับคะแนนเพื่อให้เซิร์ฟเวอร์ตรวจทานความสมเหตุสมผล พร้อมกับมีการทำ Score Hashing (นำคะแนนไปเข้ารหัสกับ Salt Key) ก่อนส่งข้อมูลออกไป เพื่อปิดกั้นการสับเปลี่ยนตัวเลขระหว่างทาง ทำให้เกมทั้งรวดเร็ว เบา และปลอดภัยในเวลาเดียวกัน

การออกแบบ UI และธีมสี (เฉพาะส่วน UI เท่านั้น)
เพื่อให้สื่อสารตัวตนของเกมอย่างชัดเจน โดยไม่แตะต้องกลไกเกมเพลย์หรือเนื้อหาอื่น ๆ ให้ใช้ระบบสีเรียบง่าย 3 ระดับดังนี้:

- สีหลัก (Main) — ขาวบริสุทธิ์: #FFFFFF
  - ใช้สำหรับข้อความหลัก หัวเรื่องขนาดใหญ่ และไอเท็ม UI ที่ต้องการความคมชัดสูง เช่น ชื่อเกม ตัวเลขคะแนนสำคัญ

- สีเน้น (Secondary) — ฟ้านีออน / Aqua Neon: #67E8F9
  - ใช้เป็นสีเน้นและแอ็กเซนต์ เช่น ปุ่มหลัก ขอบไฮไลต์ เอฟเฟกต์เรืองแสง และองค์ประกอบที่ต้องดึงสายตา
  - สามารถใช้เป็นไลต์ glow หรือ gradient ร่วมกับสีเข้มของพื้นหลังเพื่อให้รู้สึกเป็นนีออน

- สีเล็กน้อย (Little / Muted) — เทาอ่อน: #AEBBD0
  - ใช้สำหรับข้อความรอง คำอธิบาย ป้ายข้อมูลรอง เช่น คำแนะนำเล็ก ๆ วันที่ เวลา หรือคำอธิบายฟิลด์
  - ช่วยลดความเด่นเมื่อเทียบกับข้อความหลัก ทำให้ UI อ่านง่ายและไม่รบกวนภาพรวม

แนวปฏิบัติสั้น ๆ
- พื้นหลังหลักควรเป็นโทนมืด (ลึก/นวล) เพื่อให้สีขาวและนีออนเด่นขึ้น
- หลีกเลี่ยงการใช้สีอื่นเป็นจุดเน้นสำคัญ เพื่อลดความสับสนสายตาในหน้าจอที่มีเอฟเฟกต์หลายชั้น
- ให้ความสว่าง (contrast) ระหว่างข้อความหลักกับพื้นหลังอย่างน้อย 4.5:1 เพื่อความอ่านง่าย
